# FEATURES.md — 기능 카탈로그

## 테이블 스테이크 (반드시 있어야 하는 것)

### OpenAPI 스펙 생성

| 기능 | 복잡도 | 필수 여부 |
|---|---|---|
| 빌드 시 `openapi.json` 파일 생성 | 낮음 | ✅ 필수 |
| GET /openapi.json 엔드포인트 서빙 | 낮음 | ✅ 필수 |
| 모든 엔드포인트 (search, model, health) 문서화 | 낮음 | ✅ 필수 |
| 쿼리 파라미터 스키마 문서화 | 낮음 | ✅ 필수 |
| 응답 스키마 문서화 | 낮음 | ✅ 필수 |

### 클라이언트 스키마 검증 (CI)

| 기능 | 복잡도 | 필수 여부 |
|---|---|---|
| TS 클라이언트 타입 → OpenAPI spec 필드 매핑 검증 | 중간 | ✅ 필수 |
| Python dataclass → OpenAPI spec 필드 매핑 검증 | 중간 | ✅ 필수 |
| CI에서 실패 시 명확한 오류 메시지 | 낮음 | ✅ 필수 |

### GitHub Actions CI/CD

| 기능 | 복잡도 | 필수 여부 |
|---|---|---|
| `wrangler dev` 로컬 서버 실행 | 낮음 | ✅ 필수 |
| TS 클라이언트 통합 테스트 (실제 서버 대상) | 중간 | ✅ 필수 |
| Python 클라이언트 통합 테스트 (실제 서버 대상) | 중간 | ✅ 필수 |
| 기존 단위 테스트 CI 통합 | 낮음 | ✅ 필수 |

## 차별화 기능 (있으면 좋은 것)

| 기능 | 복잡도 | 판단 |
|---|---|---|
| Swagger UI (`/doc`) 서빙 | 낮음 | v2 후보 — Workers에서 CDN asset 필요 |
| Stoplight Elements UI 서빙 | 낮음 | v2 후보 |
| spec 변경 시 CHANGELOG 자동 생성 | 높음 | 범위 초과 |
| openapi-generator로 클라이언트 코드 생성 | 높음 | 명시적으로 제외 (검증만) |

## 안티 기능 (의도적으로 제외)

| 기능 | 이유 |
|---|---|
| 클라이언트 코드 자동 생성 | 기존 수동 코드 보존, 생성 툴 복잡도 추가 안 함 |
| 실시간 spec 변경 감지 | 빌드/CI 시점 검증으로 충분 |
| 다른 언어 클라이언트 추가 | 현재 범위 초과 |

## 기능 간 의존성

```
OpenAPI Zod 마이그레이션 (API)
    ↓
빌드 스크립트 (openapi.json 생성)
    ↓
    ├── TS 클라이언트 검증 스크립트
    └── Python 클라이언트 검증 스크립트
            ↓
      GitHub Actions 워크플로우
      (wrangler dev + 통합 테스트 + 검증)
```
