## 캐시 전략

API는 Cloudflare Workers Cache API를 사용하며 시간 기반 무효화 방식을 채택합니다. 캐시 키는 전체 요청 URL입니다.

| 엔드포인트 | TTL | 근거 |
|----------|-----|------|
| `GET /search` | 60초 | 새 모델이 빠르게 등장하므로 짧은 TTL로 최신 결과 유지. |
| `GET /model` | 300초 | 태그 변경 빈도가 낮으므로 긴 TTL로 업스트림 부하 감소. |
| `GET /health` | 캐시 없음 | 항상 실시간 스크래퍼 상태를 반영해야 함. |

캐시 무효화는 전적으로 시간 기반이며, 수동 무효화 메커니즘은 존재하지 않습니다.

## 에러 처리 흐름

1. **스크래퍼 계층** — `scrapeSearchPage()`와 `scrapeModelPage()`는 HTTP 실패 또는 CSS 선택자 불일치 시 `Error`를 던집니다.
2. **검증 계층** — `es-toolkit/util`의 `assert()`가 파싱 후 결과를 검증합니다 (예: 선택자가 0개 요소 반환).
3. **라우트 핸들러 계층** — 각 핸들러를 `try/catch`로 감쌉니다. 스크래퍼 에러는 적절한 HTTP 상태 코드와 함께 구조화된 `ErrorResponse`로 매핑됩니다.
4. **알림 계층** — 스크래퍼 실패 시 `AlertService`가 `Promise.allSettled`를 통해 웹훅 알림을 발송합니다. 알림 실패는 조용히 무시됩니다.

### 에러 코드

| 코드 | HTTP 상태 | 트리거 |
|------|-------------|--------|
| `INVALID_PARAMETER` | 400 | 쿼리 파라미터 검증 실패. |
| `SCRAPE_PARSE_ERROR` | 502 | HTML 파싱 실패 (선택자가 0개 요소 반환). |
| `SCRAPE_UPSTREAM_ERROR` | 502 | ollama.com으로의 요청 실패. |
| `SCRAPE_NO_RESULTS` | 502 | 스크래핑은 성공했지만 결과 없음. |
| `INTERNAL_ERROR` | 500 | 예기치 않은 서버 에러. |

## 알림 시스템

`AlertService`는 `Promise.allSettled`를 통해 구성된 모든 채널에 병렬로 알림을 발송합니다:

- **SlackChannel** — mrkdwn 포맷의 Slack 웹훅.
- **DiscordChannel** — 심각도 색상이 적용된 Discord 임베드.
- **EmailChannel** — 범용 이메일 웹훅.

웹훅 URL은 Cloudflare Workers 시크릿으로 저장됩니다 (`wrangler secret put`). `wrangler.toml`이나 버전 관리에 포함되지 않습니다.

알림 발송은 fire-and-forget 방식이며, 실패해도 요청 라이프사이클에 영향을 주지 않습니다.

## 3계층 테스트 전략

| 계층 | 범위 | 러너 | 개수 | 목적 |
|-------|-------|--------|-------|------|
| **Unit** | 개별 함수/모듈 | Vitest (Node.js) | 72 | 격리된 로직 검증. |
| **Integration** | 전체 요청/응답 체인 | 인-프로세스 Hono 또는 서브프로세스 | 39 | 직렬화, 라우팅, 스키마 정합성 검증. |
| **E2E** | 배포된 실제 API | Bash (`curl` + `python3`) | 23 | 프로덕션 환경에서의 실제 동작 검증. |

### 황금 규칙: 해로운 모킹 금지

절대 라우트 핸들러, 미들웨어, 검증 로직을 모의 서버에서 재구현하지 마십시오. 이는 프로덕션과 silently drift되는 병렬 코드 경로를 만듭니다.

**올바른 접근:** 실제 프로덕션 `app`을 import하여 `@hono/node-server`로 서빙합니다. 외부 종속성만 최하위 계층에서 스텁 처리합니다:
- 서드파티 API에 대한 `fetch()` 호출 — 응답을 메모리에 캐시.
- Cloudflare Workers 전용 글로벌 (`caches`, `ScheduledEvent`) — no-op 스텁.

### 왜 3계층인가?

| 질문 | 답변 |
|----------|--------|
| E2E만 사용하면? | 느리고, flaky하며 (ollama.com 의존), 디버깅이 어려움. |
| 유닛 테스트만 사용하면? | 직렬화 버그, 스키마 드리프트, 라우팅 문제를 놓침. |
| 통합 테스트에서 스크래퍼를 모킹하는 이유는? | ollama.com HTML 변경이 클라이언트 테스트를 깨뜨리면 안 됨. |
| E2E에서 실제 스크래퍼를 사용하는 이유는? | ollama.com이 DOM을 변경했을 때 감지해야 함. |
| E2E에 bash를 사용하는 이유는? | 종속성 제로 — `curl` + `python3`만 있는 어떤 CI 이미지에서도 동작. |

## CI/CD 파이프라인

```
Pull Request → ci.yml (unit + integration) → merge
Main push    → deploy.yml (deploy-api → health-check → e2e → publish)
```

`deploy.yml` 워크플로우는 코드 파일이 변경되었을 때만 실행됩니다 (`paths` 필터가 `*.md`와 `openapi.json`을 제외).
